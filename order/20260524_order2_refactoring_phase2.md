# 作業指示書：20260524_order2_refactoring_phase2(v0.5.04)

## 役割
あなたは優秀なフロントエンドエンジニアです。
提供したソースコードに対し、以下の「リファクタリング作業・第二段階」を行ってください。

## 目的
単一ファイル構造が抱える「高結合度」「見通しの悪さ」「メモリ管理の懸念」を解決し、今後の「物理法則の完全統合」や機能拡張に耐えうるアーキテクチャに刷新する。
**【絶対原則】ゲームの見た目、操作感、各種機能、ゲームバランスは一切変更せず、内部構造の変更のみに留めること。**

## 実装ステップと要件

### 1. エンティティの完全クラス化とメソッド分離
フェーズ1で作成した `Ship` クラス群をさらに抽象化し、画面上の全ての動的オブジェクトの最上位基底となる `Entity` クラスを定義する。
- **対象:** 弾（bullets）、ミサイル（missiles）、パーティクル（particles/explosions）、アイテム（gems）などの小物類もすべて `Entity` を継承するクラスとして再定義する。
- **メソッド分割:** 将来の物理エンジン統合に備え、全エンティティの更新処理を「思考・状態更新（`updateLogic`）」と「物理移動（`applyPhysics`）」に分割し、`update()` 内で順に呼び出す構造とする。

### 2. オブジェクトプールの導入（パフォーマンス最適化）
弾やパーティクルなど、高頻度で生成・破棄されるエンティティに対して、ガベージコレクション（GC）のスパイクを防ぐためのオブジェクトプール構造を導入する。
- **要件:** 必要な最大数をあらかじめ配列等に初期化して保持し、発生時は `new` ではなくプールから非アクティブなインスタンスを取得して再利用（初期化）する。消滅時は配列から `splice` せず、非アクティブフラグを立てるのみとする。

### 3. ハードコード（マジックナンバー）の定数化
コード内に直接記述されているUIの座標計算用数値や、ゲームバランスに関わる固定値を抽出し、一元管理する。
- **要件:** 既存の `CONFIG` オブジェクトを拡張し、`CONFIG.UI` や `CONFIG.BALANCER` などのカテゴリに分けて定数化する。描画や計算処理はこれらの定数を参照する形に書き換える。

### 4. ステートマシン（状態管理）の整理
巨大な `update()` および `draw()` 関数内に存在する、状態（TITLE, PLAYING, RESULT, LEVEL_UP など）ごとの `if...else` 分岐を廃止し、Stateパターンを適用する。
- **要件:** 各状態専用の更新関数と描画関数を持つオブジェクト（またはクラス群）に分割し、メインループからは現在の状態（`GAME.state`）に対応する処理を呼び出すだけのシンプルな構造にする。

## 段階的実行のロードマップ
バグの混入を防ぐため、IDE側で以下の順序で1タスクずつ実装・動作確認を行うこと。
1. **タスク3（定数化）:** 最も安全な数値の置き換え。
2. **タスク4（状態管理の分離）:** メインループの交通整理。
3. **タスク1（エンティティのクラス化とメソッド分離）:** データと振る舞いのカプセル化。
4. **タスク2（オブジェクトプール）:** クラス化された小物に対するメモリ最適化。
